## 1. โครงสร้างไฟล์เดิม (เป็นตาราง 2 มิติ / pivot)

**แกนคอลัมน์** = เวลา กางออกตามปีและไตรมาส
- แถว 3: บล็อกตามปี — `2024 (New 10 Unit)` = B–G · `2025 (New 10 Unit)` = H–N · `2026` = O–T
- แถว 4: ในแต่ละปีซอยเป็น Q1 / Q2 / Q3 (ไม่มี Q4)
- แถว 5: แต่ละ Q มี 2 คอลัมน์ = `S/N อุปกรณ์` + `หมายเหตุ` (ยกเว้น 2025 Q3 มีคอลัมน์พิเศษ `H/W`)

**แกนแถว** = อุปกรณ์ กางลงตาม plant
- คอลัมน์ A = Plant (พบ 7 แห่ง: ภูเก็ต, หาดใหญ่, กระบี่, สุราษฎร์ธานี, อู่ตะเภา, อุดรธานี, เชียงใหม่)
- แต่ละแถวใต้ plant = 1 ช่อง/ตำแหน่งอุปกรณ์ อ่านจากซ้ายไปขวา = ประวัติ S/N ในช่องนั้นตามรอบเวลา

## 2. กฎการอ่านข้อมูล

- **Plant เว้นว่าง** → ยึดค่าจากแถวด้านบน (ไม่พิมพ์ซ้ำ)
- **Q = รอบ PM** เข้าตรวจ 3 ครั้ง/ปี (Q1/Q2/Q3) ไม่มี Q4 — ดูตารางช่วงเวลาในหัวข้อย่อยถัดไป
- **Legend สี (แถว 1):** เขียว = ส่งมอบเครื่องใหม่ · ส้ม = เสื่อมสภาพ · เหลือง = ยืม/ถูกยืม
  - สีลงไว้ที่เซลล์ S/N แต่ลงไม่ครบทุกเคส → **ยึด "ข้อความหมายเหตุ" เป็นแหล่งความจริง** แล้ว derive ประเภท/สีใหม่เอง
- **แถว 2 (prefix → ปี):** RP9 = 2025 · RNC/RPB = 2023 · RMA = 2021 · RJB = 2018
  - ตีความว่าเป็นปีของรุ่น/การส่งมอบ ตาม prefix ของ S/N (ใช้บอกอายุเครื่องได้)
- **คอลัมน์ H/W:** เป็นหมายเหตุระดับ **plant + Q** (merge คลุมหลายแถว) ไม่ใช่ระดับเครื่อง — ยืนยันแล้ว

### ประเภทหมายเหตุที่พบ (ใช้ group เป็นสถานะใหม่)
| ข้อความในไฟล์ (ตัวอย่าง) | ประเภทสถานะ |
|---|---|
| ส่งมอบเครื่องใหม่ / ส่งมอบใหม่ 25xx | รับเครื่องใหม่ (New delivery) |
| เสื่อมสภาพนำกลับ Enco | เสื่อมสภาพ/คืนผู้ผลิต (Retired) |
| ยืมจาก <plant> / ยืมจาก สอ.<plant> | ยืมเข้ามาใช้ (Borrowed-in) |
| สอ.<plant> ยืมใช้งาน / ยืมใช้ | ถูกยืมออกไป (Loaned-out) |
| แก้ไขจากเลข <S/N> | แก้ไขข้อมูล (Correction) |

### ช่วงเวลาของรอบ PM (ยึดจากแผน 2025)

| Q | ช่วงงานภาคสนาม | กำหนดส่งเอกสาร |
|---|---|---|
| Q1 | ต้น ก.พ. – กลาง มี.ค. | 31 มี.ค. |
| Q2 | ปลาย พ.ค. – กลาง ก.ค. | 31 ก.ค. |
| Q3 | กลาง ก.ย. – ต้น พ.ย. | ~21 พ.ย. |

**ช่วงไร้ PM** (ทีม on-site ไม่เข้า plant):
- ปลาย พ.ย. → ต้น ก.พ. ปีถัดไป (ใหญ่สุด ~2.5 เดือน) ← ที่เคยเรียก "Q4"
- เม.ย. และ ส.ค. (ระหว่างรอบ — มักทำงาน Amazon/วังน้อย/CM/เอกสาร)

เหตุผลที่เลี่ยงปลายปี: ต้องล่าลายเซ็นผู้เกี่ยวข้อง + ลูกค้ามีเส้นตายวางบิล

วันปิดรอบจริงอาจขยับ 1-2 สัปดาห์แต่ละปี → ระบบควรเก็บเป็น **config ต่อปี** (ไม่ hard-code)

## 3. การตัดสินใจหลัก: ยึด "ตัวเครื่อง" เป็นหัวใจ (Device-centric)

**ปัญหาที่ต้องตัดสิน:** 1 แถว ≠ 1 เครื่องเสมอ และ 1 เครื่อง ≠ 1 แถวเสมอ
- *แถวเดียวมี 2 เครื่อง:* แถว 15 ภูเก็ต — RJB03F0088 ถูกแทนด้วย RPB03T0403 ใน 2024 Q3
- *เครื่องเดียวอยู่ 2 plant:* RPB03T0409 เป็นของอุดรธานี แต่กระบี่ยืมไปใช้ (โผล่ทั้งสองแถว)

**ตัดสินใจ:** ระบบยึด **เครื่องจริงตาม S/N** เป็น entity หลัก ส่วน "ตำแหน่งที่ plant" และหน้าตาตาราง Excel เดิม จะ *คำนวณกลับ* จากข้อมูลเครื่องได้

**เหตุผล**
1. จะมีอุปกรณ์อีกหลายชนิด → ยึด S/N ขยายได้สะอาดกว่า
2. เคสยืม/ถูกยืมข้าม plant จัดการเป็น "เหตุการณ์ของเครื่อง" ได้ตรงไปตรงมา
3. ตรงกับ use case หลัก (ดูข้อ 4)

## 4. Use case หลัก ที่ขับเคลื่อนการออกแบบ

> สถานการณ์: field engineer **ถือเครื่องอยู่ในมือ** แล้วอยากรู้เรื่องเครื่องตัวนั้น

| คำถาม | ข้อมูล/ความสัมพันธ์ที่ต้องมี |
|---|---|
| เครื่องนี้เป็นของ plant นี้ไหม | plant เจ้าของ (home) เทียบ plant ปัจจุบัน |
| ยืมมาจากใคร | ความสัมพันธ์ยืม → plant ต้นทาง + ช่วงเวลา |
| รับมาใหม่ไหม | เหตุการณ์รับเครื่องใหม่ + ปีที่รับ |
| ทดแทนเครื่องไหน | ลิงก์ "แทนเครื่อง S/N เก่า ที่ plant ไหน รอบไหน" |
| warranty หมดเมื่อไร | วันหมดประกัน + ใครรับผิดชอบ PM (vendor/on-site) — ดูข้อ 6 |

## 5. ร่างโครงสร้างข้อมูลเบื้องต้น (ยังไม่ final — ไว้คุยรอบหน้า)

**Device (เครื่อง — ตัวหลัก)**
- `s_n` (รหัสหลัก), `asset_type` (Tablet…)
- `prefix → ปีรุ่น` (derive จาก S/N prefix)
- `home_plant`, `current_plant`
- `received_q`, `received_year` (Q+ปีที่รับเครื่อง)
- `warranty_months` (default 12 — ตั้งต่อ asset_type ได้)
- `warranty_expiry` (derived — ดูสูตรในข้อ 6)
- `pm_responsibility` (derived: `vendor` หรือ `on_site`)
- `current_status`
- `replaced_sn` (เครื่องเก่าที่ถูกแทน)

**PM Snapshot (บันทึกแต่ละรอบ — ใช้สร้างประวัติ & ตารางเดิม)**
- s_n, year, quarter(Q1–Q3), plant_ที่อยู่, status_category, note_raw, hw_note(ระดับ plant)

**Relationship**
- Borrow: เครื่อง ↔ plant ที่ยืมไป + ช่วงเวลา
- Replacement: เครื่องใหม่ ↔ เครื่องเก่า ที่ plant + รอบที่เปลี่ยน

## 6. Warranty & ความรับผิดชอบ PM (เชื่อมโยงกัน)

### Insight สำคัญ
Warranty ในบริบทนี้ **ไม่ใช่** แค่ "ของพังซ่อมฟรี" แต่เป็นตัวกำหนด **"ใครเป็นคน PM"**
- เครื่อง **ในประกัน** → vendor/บริษัทเจ้าของอุปกรณ์เข้ามา PM (ทีม on-site ไม่ต้องไป)
- เครื่อง **หมดประกัน** → ทีม on-site รับช่วงต่อ → เพิ่มเข้า PM coverage list ของเรา → กระทบเวลา/แผนเดินทาง

→ Warranty จึงเป็น **input ของการวางแผน PM** ไม่ใช่แค่ข้อมูลโชว์

### แหล่งข้อมูล warranty
ไม่มีระบบทะเบียนประกันแยก → คำนวณจากข้อมูลที่มี:
- ถ้ามีบันทึก "ส่งมอบเครื่องใหม่" → ใช้ `Q+ปี` ที่รับเป็น anchor
- ถ้าไม่มี (เครื่องก่อน 2024) → ใช้ prefix→ปี เป็น fallback (สมมติ Q1 ของปีนั้น)

### สูตรคำนวณ
```
warranty_start = วันปิดรอบของ Q ที่รับ
                 (Q1→31 มี.ค., Q2→31 ก.ค., Q3→~21 พ.ย. — config ต่อปี)
warranty_end   = warranty_start + warranty_months  (default 12)
```

### Field ที่ derive ได้
- `pm_responsibility`: `vendor` ถ้า warranty ยังคุ้มถึง "วันเริ่มรอบ PM ถัดไป", `on_site` ถ้าหมดก่อนหน้านั้น
- `transition_at_next_round`: `true` ถ้าจะเปลี่ยนจาก vendor → on_site ในรอบถัดไป (ต้องเพิ่มเข้า coverage)

### กฎ Alert (ใช้ตำแหน่ง PM รอบเป็นตัวกำหนด ไม่ใช้จำนวนวันตายตัว)
> ระบบมองไปยัง **วันเริ่มรอบ PM ถัดไป** → เครื่องไหน warranty หมดก่อนหน้านั้น flag เป็น `transition_at_next_round` ทันที

ตัวอย่าง:
- เครื่องหมดประกัน 15 ธ.ค. 2025 (อยู่ในช่วงไร้ PM)
- รอบ PM ถัดไป Q1 2026 เริ่ม ก.พ. 2026
- 15 ธ.ค. 2025 < ก.พ. 2026 → flag ตั้งแต่ปิด Q3 2025 (พ.ย.) ว่า "ต้องเพิ่มเข้า Q1 2026 coverage list"

กฎเดียวจัดการเคส "หมดในช่วงไร้ PM" ได้อัตโนมัติ

### มุมมองใหม่ที่ unlock ได้
**"PM Coverage List"** ต่อ plant ต่อรอบ:
- รายการอุปกรณ์ที่ทีม on-site ต้อง PM ในรอบนั้น (`pm_responsibility = on_site`)
- + รายการที่จะ "เข้ามาใหม่" จากการหมดประกัน (`transition_at_next_round = true`)
- ใช้ประมาณเวลา/วางแผนการเดินทางในแต่ละรอบ

## 7. ขอบเขตการใช้งาน, User Roles & Tech Stack

### ลูกค้า
**ลูกค้าเดียว: OR** → ไม่ทำ multi-tenant, schema ไม่มี `customer_id`

### Data retention
เก็บประวัติย้อนหลังสูงสุด **5 ปี × 3 รอบ = 15 รอบ PM** ต่อเครื่อง

### 3 User Roles

| Role | สิทธิ์ | งานหลัก |
|---|---|---|
| `customer` (OR) | read-only | ดู asset ปัจจุบัน + history ย้อนหลังสูงสุด 15 รอบ |
| `support` | write | บันทึก status ของแต่ละรอบ PM ให้ข้อมูลทันสมัย |
| `inspector` *(future)* | write | กรอก checklist (ใช้งานปกติ/ส่งซ่อม) + หมายเหตุ ครอบคลุมหลายชนิดอุปกรณ์ |

**Design insight:** Role `support` กับ `inspector` ทำงานบน entity เดียวกัน (= PM Snapshot ของรอบ) แค่ `inspector` มี field เพิ่ม → ออกแบบ form แบบขยายได้ ไม่ต้องสร้างระบบใหม่ตอน Role 3 มา

### Tech Stack
- **Backend/Framework:** Django (Python)
- **Database:** SQLite สำหรับ v1 → ย้าย Postgres ทีหลังแทบไม่กระทบ code
- **Data Entry สำหรับ support:** ใช้ **Django Admin** built-in → ประหยัดเวลาเขียนฟอร์ม
- **Frontend:** Django templates เรียบ ๆ (เสริม HTMX/Alpine ทีหลังถ้าต้องการ interactivity)
- **Auth:** Django built-in user / group / permission ครอบคลุม 3 role ได้

### เหตุผลที่ไม่เลือกตัวอื่น
- **Streamlit/FastAPI**: ต้องเขียน user/role/admin form เอง — กินเวลา
- **Next.js/full-stack JS**: เกินตัวสำหรับ scale ปัจจุบัน + ผู้ดูแลไม่ใช่ full-time dev
- **Django**: สมดุลระหว่าง "ของพร้อมใช้" กับ "เข้าใจ/บำรุงรักษาได้"

### ประเด็นรองที่ยังไม่ตัดสิน (ไม่กระทบ schema)
- ทีม support แก้รอบเก่าได้ไหม หรือ lock หลังปิดรอบ
- ลูกค้าเห็น real-time หรือต้องรอปิดรอบ

ค่อยตัดสินตอน design UI/workflow ก็ได้
